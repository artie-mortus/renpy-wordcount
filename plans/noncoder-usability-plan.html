Product and implementation plan

Make Say Count feel like a writing app that quietly produces Ren’Py

A phased plan for welcoming nontechnical writers, reducing visible complexity, teaching through actions, and preserving the editor’s advanced capabilities and safety guarantees.

Executive summary

Recommended direction: keep Ren’Py code as the durable source of truth today, but place a friendly welcome experience, contextual actions, natural-writing conversion, guided fixes, and visual insertion tools in front of it. Preserve technical commands under menus and command search.

The current application has excellent expert breadth but exposes six permanent toolbar actions, Ren’Py terminology, raw code on the empty screen, and project concepts before a new writer has formed a mental model. Most required foundations already exist: manuscript conversion with preview and undo, diagnostics and repairs, project discovery, workspace restoration, snapshots, an asset insertion pattern, and extensive core tests.

The main architectural constraint is that prose conversion is one-way. A cosmetic “Writing / Code” toggle would imply lossless round trips that the app cannot provide for arbitrary Python, conditions, menus, transforms, and definitions. Phase 2 should instead offer Write naturally → Review → Make game script. A persistent dual-view writer model is a separate Phase 5 investment.

Product principles and target flow

Use writer language first

Say “Open a game,” “Problems,” and “Version history.” Reveal “project folder,” “diagnostics,” and Git terms only where precision is necessary.

Progressive disclosure

The primary surface shows the next useful action. Expert commands, shortcuts, pane names, and stable IDs remain available.

Preview meaningful changes

Project creation, prose conversion, repairs, and generated snippets must preview or preflight their effects and remain undoable or recoverable.

Never overstate safety

Distinguish “saved to file” from app-local version history and from off-device backup or Git synchronization.

Audit findings

P1

Startup has no true home state

EditorNotebook always creates scene-1.rpy before MainFrame::RestoreWorkspace(). The editor overlay in src/ui/editor_notebook.cpp is static text with a raw Ren’Py example, not actionable onboarding.

Recommendation: create a separate WelcomePanel and explicitly resolve restored-workspace-versus-welcome lifecycle. Do not simulate a full welcome screen inside Scintilla.

P1

Primary actions and labels are hardcoded in multiple places

BuildCommandBar(), BuildMenus(), and OnCommandPalette() duplicate labels. The command bar permanently shows Open, Save, Convert, Production, Git, and Run.

Recommendation: centralize user-visible command metadata, retain stable IDs, and add RefreshCommandBarState(). Keep AUI pane .Name() values stable so saved layouts continue to restore.

P1

“Start a new story” needs real project creation

src/core/project.cpp discovers existing folders only. A loose script is not necessarily a playable Ren’Py game.

Recommendation: add a pure, tested project-creation service with validation, collision preflight, explicit templates, and recoverable writes. Never overwrite existing files silently.

P2

Natural writing is mature but not a persistent mode

src/core/manuscript.* and ManuscriptReviewDialog already provide conservative conversion, uncertainty review, character reuse, and undo. Choices and the proposed stage-direction syntax require explicit parser support and tests before the UI promises them.

P2

Problems can be friendlier

Diagnostics already have typed ranges, jump navigation, and a snapshot-protected bulk repair. Messages can expose syntax jargon; per-row repairs lack stable fix IDs.

Recommendation: map diagnostic types to a plain primary explanation, optional technical detail, and stable action metadata. Never dispatch a repair by matching English copy.

P2

Safety exists but is mostly invisible

Atomic save, dirty tab markers, ten-minute automatic snapshots, pre-mutation snapshots, and restore already exist. MainFrame receives no document dirty/save callback, and “Snapshots” reads as technical.

Recommendation: add document-state events and display accurate “Unsaved changes,” “Saved locally,” and “Version saved” states. Relabel only the UI; keep storage formats/types stable.

Recommended implementation phases

Phase 1 · P1

Plain-language shell and contextual hierarchy

  • Create one command metadata source for menu, command palette, and primary labels/descriptions.
  • Use “Open a game,” “Problems,” “Check game for problems,” and “Version history” on beginner-facing surfaces. Keep accurate Git terminology inside setup and errors; “Back up & sync” must not imply automatic cloud backup.
  • Store command-bar controls as members and refresh them for: no project, loose script, connected project, dirty/saved, fixable problems, Ren’Py unavailable/configured/running.
  • Add an EditorNotebook document-state callback. Unify Save and Save As completion so status/history promises are consistent.
  • Do not rename internal AUI pane identifiers or command IDs.

Likely files: src/ui/main_frame.{h,cpp}, src/ui/editor_notebook.{h,cpp}, src/ui/snapshot_dialog.cpp, possibly new src/ui/commands.{h,cpp}.

Acceptance: at 820×560 and 200% scaling, the primary bar does not overflow; keyboard focus is visible; Save status distinguishes disk save from local history; all advanced commands remain reachable.

Phase 2 · P1

Welcome, onboarding, and safe project creation

  • Add WelcomePanel with Start a new story, Open a game, Open a script, and recent games.
  • Change startup sequencing: restore valid workspace first; show welcome only when there is no usable restored content. Avoid startup flash and disposable scene-1.rpy.
  • Add a versioned onboarding state with backward-compatible missing-key defaults. Derive checklist milestones from real events where possible.
  • Add core/project_creation.* and NewProjectDialog. Separate pure planning/preflight from filesystem application; define rollback/collision behavior.
  • Offer “Start new writing” as a natural-language draft that explicitly becomes Ren’Py through the existing review flow. Do not label it a reversible mode.

Likely files: new src/ui/welcome_panel.*, src/ui/new_project_dialog.*, src/core/project_creation.*; changes to main_frame, editor_notebook, app/settings, CMake, and tests.

Acceptance: restored users never lose or bypass their workspace; project creation never overwrites; first-time users can create/open, write dialogue, review conversion, save, and preview using keyboard-only navigation.

Phase 3 · P2

Guided problems and contextual repair

  • Add plain explanations and optional “Technical details.”
  • Introduce stable diagnostic/fix IDs and targeted core repair functions.
  • Surface “Fix N problems” only when actionable; show preview, cancel, apply, undo, and snapshot protection.
  • Keep ambiguous diagnostics advisory; do not broaden auto-fix heuristics merely to increase the count.

Likely files: src/core/diagnostics.*, src/ui/diagnostics_panel.*, main_frame, diagnostic tests.

Phase 4 · P2

Visual Insert tools

  • Add an Insert menu/button for character, dialogue, choice, background/scene, music/sound, and jump.
  • Reuse AssetPanel’s typed insertion-handler pattern.
  • Put generation and validation in core/script_builder.*; UI dialogs only gather fields and preview output.
  • Validate escaping, caret indentation/context, duplicate aliases/labels, and jump targets. Definitions may need insertion into a known definitions file instead of at the caret.

Acceptance: every insertion is previewable and one-step undoable; generated code parses; invalid context gives a plain next action.

Phase 5 · Defer pending design

Persistent writer-first document model

Only build a Writing/Code dual view after deciding how manuscript source, generated script, manual code edits, unsupported constructs, workspace restoration, export, and conflicts coexist. This likely requires an AST or source map plus explicit unsupported-code representation.

Do not: auto-convert on every keystroke/save, store natural prose as temporarily invalid .rpy without a clear contract, or imply lossless round trips.

Test and verification strategy

The current build registers 224 tests. Core coverage is broad, but the only dedicated wx UI target is say-count-chat-ui-tests; it does not instantiate MainFrame. Add say-count-main-ui-tests or isolate new components/state reducers so they can be tested without the whole frame.

  • Automated UI: welcome actions/copy/focus order; restored workspace precedence; contextual command states; onboarding persistence; save/history indicators; dialog cancel/apply behavior.
  • Core: project creation preflight/apply/rollback; settings backward compatibility; diagnostic action IDs; builder escaping/indentation; manuscript choice/stage-direction extensions.
  • Manual accessibility: keyboard-only flow, focus restoration after state changes, screen-reader names/roles/status announcements, 200% scaling, light/dark/system/high-contrast behavior, and no color-only meaning.
  • Regression: workspace restore, dirty prompts, external conflicts, snapshots, manuscript preview/undo, project discovery, command shortcuts, and AUI layout restoration.

Decisions required before implementation

  • Does “Start a new story” create a complete Ren’Py project, a Say Count writing draft, or offer both?
  • Should project creation call an installed Ren’Py SDK’s canonical generator or own a minimal versioned template?
  • Is welcome shown only on first launch, whenever the workspace is empty, or as a reusable Home page?
  • Where do natural-writing drafts live: .txt/.md, a dedicated format, or project sidecars?
  • Which checklist milestones are useful without feeling like mandatory setup?
  • Which beginner-facing Git promise is truthful across repositories with no remote or credentials?

Do not change yet

  • Do not change workspace.dat positional format without an explicit version/migration.
  • Do not remove technical menus or shortcuts; simplify the primary surface.
  • Do not rename stable AUI pane names used in saved layouts.
  • Do not bypass manuscript review, Undo, snapshot protection, external conflict handling, or dirty-close prompts.
  • Do not mix wxWidgets concerns into core conversion/project-generation modules.
  • Do not call app-local history “cloud backup.”

Suggested approval boundary

Approve and implement Phase 1 only first. It offers immediate noncoder value with low data-model risk and produces the document-state/command-state seams needed by the welcome experience. Review its screenshots and keyboard flow before beginning Phase 2.
